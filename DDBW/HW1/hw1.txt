Домашнее задание: Проектирование хранилища данных для медицинской отрасли
1. Общая постановка задачи

Спроектировать архитектуру хранилища данных (DWH) и аналитических витрин (Data Marts) для сбора и анализа информации из лечебно-профилактических учреждений (ЛПУ). Система должна служить источником данных для аналитических панелей (дашбордов) для ключевых стейкхолдеров.

2. Конкретные задания по методологиям

Необходимо спроектировать три различные модели хранения данных, каждая из которых решает общую задачу, но с использованием разных подходов.

Задание 2.1: Методология Инмона (Централизованное хранилище)

Задача: Спроектировать единую реляционную базу данных в 3-й нормальной форме (3НФ).

Требование: Эта БД должна служить централизованным хранилищем ("единым источником правды") для всех данных, описанных в разделе "Бизнес-требования". Она будет являться источником для наполнения аналитических витрин.

Задание 2.2: Методология Кимбалла (Витрины данных)

Задача: Спроектировать три аналитические витрины данных, используя архитектуру "звезда" (таблица фактов + таблицы измерений).

Требование: Витрины должны быть спроектированы для ответа на конкретные аналитические запросы (см. раздел "Требования к витринам данных").

Задание 2.3: Методология Data Vault

Задача: Спроектировать единую реляционную базу данных в соответствии с методологией Data Vault.

Требование: Модель данных должна состоять из ключевых компонентов Data Vault: хабов (Hub), линков (Link) и сателлитов (Satellite) и охватывать все сущности и процессы из раздела "Бизнес-требования".

3. Требования к витринам данных и аналитике

Необходимо спроектировать витрины, которые позволят строить следующие дашборды:

Свободные койки:

Что показывает: Текущий статус по загруженности койко-мест в больницах (по состоянию на "вчера").

Фильтры: По ЛПУ, по типу палаты (ВИП/стандартная).

Расписание врачей:

Что показывает: Доступные для записи слоты в расписании врачей на текущий месяц.

Фильтры: По ЛПУ, по профилям врачей.

Динамика обращений:

Что показывает: Динамику обращений пациентов к врачам.

Гранулярность: Неделя.

Фильтры: По ЛПУ, по профилям врачей.

4. Бизнес-требования к данным и процессам

Хранилище должно поддерживать следующий бизнес-процесс и хранить указанную информацию.

Пациент:

При обращении в ЛПУ на пациента заводится "амбулаторная карточка".

Необходимо хранить персональные данные пациента.

Врач:

У врача может быть один или несколько профилей (специальностей).

Врач может работать в одном или нескольких отделениях одного ЛПУ.

Врач может работать в нескольких разных ЛПУ.

Необходимо хранить персональные данные врача.

ЛПУ (Лечебно-профилактическое учреждение):

Необходимо хранить полную информацию об ЛПУ: адрес, профиль, телефон и т.д.

В ЛПУ есть палаты и койко-места. Палаты бывают двух типов: ВИП (4 койко-места) и стандартные (8 койко-мест).

Процесс: Посещение врача:

Должен сохраняться факт каждого посещения пациентом врача.

По результатам посещения пациенту ставится диагноз.

Назначается лечение: амбулаторное (дома) или стационарное (госпитализация).

Процесс: Госпитализация:

Госпитализация возможна только по направлению от врача после визита.

При госпитализации пациент занимает койко-место.

Необходимо хранить факт нахождения в стационаре, включая диапазон дат (с какого по какое число).

Система должна позволять определять, какие койко-места свободны, а какие заняты.

Процесс: Расписание и запись на прием:

Запись к врачу возможна только в свободные слоты в его расписании.

Необходимо хранить информацию о расписании приемов врачей.

5. Требования к справочным данным

Система должна обеспечивать возможность хранения и использования следующих справочников:

Профили врачей (например, "Терапевт", "Хирург").

Профили ЛПУ (например, "Многопрофильная больница", "Поликлиника").

Отделения ЛПУ (например, "Терапевтическое", "Кардиологическое").

Международный классификатор болезней (МКБ).

КЛАДР (Классификатор адресов России).

6. Глоссарий

ЛПУ – лечебно-профилактическое учреждение (поликлиника, больница, медцентр и т.д.).
